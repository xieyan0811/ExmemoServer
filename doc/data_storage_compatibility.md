# 数据存储重构与过渡期兼容设计 (Data Storage & Compatibility)

## 1. 背景与目标
在将其核心的数据处理底座 `dataforge` 从原先笨重的 Django (exmemo) 迁移至新型基于 FastAPI 的 `ExmemoServer` 时，我们确立了**“文件即源 (File as Source-of-Truth)”**的全新数据存储架构。

未来的终态是：
- 所有笔记及媒体数据持久化存储在独立的对象存储 (MinIO) 中，不再直接依赖宿主机文件系统。
- 砍除原本繁琐的文件切块 (Block) 及强依赖的向量化机制。
- PostgreSQL 数据库从“主存储”降级为“索引库 / 缓存”，仅包含指向 MinIO 对象的路径 (`path`)和基本搜索元数据。即便数据库损坏，也可以直接扫描庞大的 MinIO 数据桶来无损重建数据库。

## 2. 痛点：过渡期的多系统共存问题
彻底删除老旧字段和机制会导致目前仍在服役的旧 Django 前端 / Web 端瞬间崩溃，因为旧代码强依赖数据库里特定的分块结构（如 `block_id` 和 `raw`）去捞取数据。在此割接换型的过渡阶段，必须要保证**底层数据不动、表结构不破坏、双端无缝互通**。

## 3. 过渡期解决方案：“双轨制”与双写兼容 (Dual-Write)

为了照顾老系统的逻辑，目前 `ExmemoServer/dataforge/crud.py` 中实施了如下妥协兼容策略：

### 3.1 写入机制 (Create)
当从 `ExmemoServer` 接收到存储请求时：
1. **纯净文件落盘**：首先将传入的文本按规则序列化为 Markdown，并上传至 MinIO 的 `<类型>/<年份>/<月份>/<UUID>.md` 路径中。
2. **模拟老系统刷库**：
   - 生成 1 条 `block_id = 0` 的记录保存至 `store_entry` 表，其中 `raw` 保存截断的前 200 字内容（充当摘要）。
   - 生成 1 条 `block_id = 1` 的记录保存至 `store_entry` 表，其中 `raw` 保存完整文本。
   - 在这两条记录中，都会写入最新的 `path` 字段，用于标记该文件在 MinIO 的真实位置。

*结果*：由于完全复现了 `block_id` 生成逻辑，并填充了 `raw` 字段，旧版 Django 系统拉取数据时完全感受不到“其实背后是由新架构存的”，完美兼容展示。

### 3.2 读取机制 (Read Fallback)
为了保证 ExmemoServer 可以无缝读取旧笔记和新笔记，我们在读取逻辑处做了降级(Fallback)：
- **如果有 `path`**：说明是进入新架构后创建/修改的节点，系统会优先去 MinIO 对应路径提取高清完整的 Markdown 数据返回。
- **如果没有 `path`**：说明是历史遗留问题（只存在于旧数据库，且没有同步进文件系统），此时系统会自动 fallback，去老字段 `raw` (且 `block_id = 1` 的主记录) 里面抓取内容展示。

### 3.3 修改与删除机制 (Update / Delete)
- **更新**：先同步更新 MinIO 文件，然后去数据库中捞出 `block_id = 0` 及 `1` 这两根共生记录，将其内部的 `raw`、时间戳和附带的 `path` 同步刷新。
- **删除**：不仅从 MinIO 删除实体文件，同时也会将关联此 UUID 的所有子块 (`block_id`) 统一软删除 / 硬删除。

## 4. 后续演进计划 (Deprecation Plan)
这套双写机制仅仅是**临时脚手架**。
我们后续的演进路线应为：
1. 完成 `exmemo` (Django 端) 所有周边功能的重构及剥离。
2. 彻底停用旧版 `exmemo` 的后端进程。
3. 编写一个洗数脚本：将历史上所有仅有 `raw` 没有 `path` 的老数据批量转换为 Markdown 刷入 MinIO。
4. 在数据库中 `DROP` 掉冗余的 `raw`、`block_id`、`embeddings` 等字段，将 `models.py` 彻底精简。
5. 删掉这一套因为妥协而写下的冗杂的双写代码。
