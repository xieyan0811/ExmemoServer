# 核心模块重构与迁移设计 (Core Refactoring Design)

## 1. 重构背景与目标
基于“Exmemo 知识管理系统”的全局架构规划，我们需要将原有跑在 `exmemo/backend` (Django) 项目里的业务逻辑进行一次“去粗取精”的重构：把业务核心抽离出来，并入全新的统一坚实后端 `ExmemoServer` 之中。

本次重构的唯一标准是：**死抱知识管理的主线，沉淀数据底座**。
一切与“笔记存储、输入输出、关联思考”紧密结合的模块，一律迁入或在 `ExmemoServer` 中重写强化；所有外围插件一概剥离核心主链路。

## 2. 迁入 ExmemoServer 的四大核心基座
根据梳理，现有系统中有且仅有以下四个模块属于系统的“绝对核心”，它们将共同组成新后端的数据与逻辑骨架：

### 2.1 数据存储与结构基座：`app_dataforge`
*   **原有职责**：基础数据存储。
*   **新架构定位**：系统的“物理存储内核”，知识管理的数据底座。
*   **重构方向**：在 `ExmemoServer` 中，它将负责所有结构化知识节点（无论是 Obsidian 的长文笔记，还是手机录音转换的闪念碎片，亦或是阅读产生的高亮）的统一模型定义（Data Modeling）和持久化存储机制。

### 2.2 笔记交互与 Obsidian 同步引擎：`app_sync`
*   **原有职责**：针对 Obsidian 的笔记同步。
*   **新架构定位**：知识图谱的 API 网关与版本控制器。
*   **重构方向**：
    1.  提供最核心的系统级 CRUD（增删改查）与高级搜索服务，供任意合法的前端接入调用（App, Web, UI-less 语音端等）。
    2.  接管对本地 Obsidian 目录/文件的双向同步与变更合并（Merge）控制，确保多端输入时的数据原子性与一致性。

### 2.3 辅助思考与智能引擎：`app_ai`
*   **原有职责**：AI 核心操作。
*   **新架构定位**：系统的“左脑（逻辑整理）与右脑（语音感知）”。
*   **重构方向**：它将与 `ExmemoServer` 现有的 `asr` / `llm` / `record` 目录机制合并。
    1.  **感知输入**：通过强大的 ASR 模型（接收 ExRecord 的语音）与 TTS 输出，承载无界面语音交互的 I/O。
    2.  **知识内化**：调用 LLM 异步对存入 `app_dataforge` 的长短笔记进行二次加工——自动提取标签、摘要聚合、发现笔记之间的逻辑关联，产生能被 `app_sync` 推送到 Obsidian 的总结性反思。

### 2.4 底层任务与异步调度队列：`user_tasks`
*   **原有职责**：比较底层，用于数据同步和其他任务。
*   **新架构定位**：核心模块运转的传送带。
*   **重构方向**：在 `ExmemoServer` 引入健壮的后台任务机制（如基于 Celery, RQ 等）。因为 LLM 的大模型推理生成、ASR 的大体积语音文件转写、以及与 Obsidian 产生的大批量同步 IO，都绝不能堵塞前端（例如录音 App）的交互请求，必须强依赖 `user_tasks` 模块在后台有条不紊、具备一定容错和重试机制地执行。

## 3. 重构红线原则 (Red Line)
*   **强解耦**：上述四大核心在 `ExmemoServer` 中应相互独立调用（如 `app_record` 只管接收语音并通过 `user_tasks` 发给 `app_ai`，处理完后再存给 `app_dataforge`）。
*   **拒接插件入核**：曾经的页面抓取书签(`app_bm_keeper`)、饮食记录(`app_diet`)、翻译(`app_translate`)等一律作为外部组件，通过标准的微服务或 OpenAPI 对接，绝不污染这里的核心数据库表结构设计。

## 4. 实施策略与分步计划
为了避免步子迈得太大导致系统停摆，采用**“绞杀者架构 (Strangler Fig Pattern)”**进行极其平滑的过渡：新旧系统并存，共用一个数据库。

### 第一步：连通共用数据底座 (当前阶段的核心)
这是最关键、也是收益最快的一步。**不迁移数据、不动老代码、新旧共存。**
*   **动作**：在 `ExmemoServer` 中引入 FastAPI 和 SQLAlchemy。参照老 Django 项目 `app_dataforge` 里的 `StoreEntry` 模型，在 SQLAlchemy 中写一个同名的映射类，**强制绑定到现有的 PostgreSQL `store_entry` 表上**。
*   **效果**：`ExmemoServer` 获得对现有知识库库表直接的读写能力（增删改查）。旧的 Django 系统和新的 FastAPI 系统像两扇并排的门，操作同一个“数据仓库”。
*   **短期产出**：立马提供一套全新的、干净的上传 API（如 `/api/upload`），把 `ExRecord` 语音转写的文本直接塞进数据库，网页端一刷新就能看到。新前端接新接口，老网页走老逻辑，互不干扰，完全无缝兼容。

### 后续步骤预告：
*   **第二步**：在新后端补齐 Obsidian 的同步逻辑（平替原有 `app_sync`），使得笔记的修改和电脑端双向打通。
*   **第三步**：引入后台异步任务流，让 `ExmemoServer` 接管原有的 AI 处理链路，自动为新灌入的数据打标签。
*   **第四步**：完全架空旧 Django 后端，移除废弃代码，完成数据与服务的彻底迁移。
