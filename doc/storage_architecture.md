# Exmemo 存储架构设计

## 核心理念：双轨模式 (Dual-Track Architecture)

目前系统确立了 **“MinIO/S3 当作数据之源 (Source of Truth) + 数据库当作索引树 (Index/Cache)”** 的核心存储架构。

### 1. MinIO 成为主角（Source of Truth）
- **定位**：永远不会丢失的数据底表，业务的绝对核心。
- **内容形态**：文章的正文以及关联元数据。未来规划将直接把文章的正文与基于 Markdown 的 Frontmatter（YAML 头数据，用来替代/补充 S3 Metadata，更方便人类直接阅读）存成稳固的文件体系。
- **优势**：最高级别的数据安全与可迁移性。即使脱离了当前系统，用户也能用任意文本编辑器打开文件阅读。一旦发生灾难性故障，只要对象存储里的文件还在，整个系统的数据就可以通过扫描文件重新构建。

### 2. 数据库退化为辅助索引（Index / Cache）
- **定位**：基于文件系统提取出的一层“快捷方式目录”，用于补足对象存储的短板。
- **机制与场景**：
  - **列表与查询加速**：当用户在 App 中滑动列表、搜索关键字、按分类或时间筛选时，直接访问关系型数据库，利用 B-Tree 倒排索引实现毫秒级响应，避免了对象存储遍历文件和读取元资料时的 O(N) 性能灾难。
  - **按需加载重内容**：当且仅当用户点开某一篇具体长文章/录音时，服务端或客户端才会通过数据库里存储的 `path` 去 MinIO 里将原文件（Payload）拉取出来进行展示。


## 架构决策探讨：多端生态下的存储访问模式

在考虑是否让客户端直接接入底层文件存储服务（S3/MinIO）还是通过集中式服务器（ExmemoServer）代理请求时，我们面临以下两种模式的抉择：

### 模式 A: Local-First 直连 S3 模式 (去中心化)
- **机制**：前端 App/工具配置云厂商的 `Endpoint`/`AccessKey`/`SecretKey`，直接向 S3 或 MinIO 发起 `GET/PUT` 操作，依靠前端建立本地的 SQLite 等轻量数据库同步索引。
- **优点**：
  - “零后端”维护：只依赖极高可用性的云存储即可，非常轻量。
  - 终极隐私：数据不经过任何中间服务器，"自带密钥 (BYOK)"。
- **代价**：
  - **失去统一的 AI 等处理能力**：数据落盘后无法自动触发后端的向量化 (Embeddings)、语音识别、自动分类整理。

### 模式 B: Centralized Hub 统一服务器模式（当前演进架构推荐）
- **机制**：网页端、安卓 App、微信插件、Obsidian 插件全部作为轻量级胖客户端接入 `ExmemoServer` 的聚合 API，依靠后端发起对 MinIO 和 数据库的调度。
- **优点**：
  - **维持多端一致性**：在前端极多的情况下（Web、App、微信机器人、Obsidian 等），把复杂的索引逻辑、缓存逻辑、搜索逻辑实现在后端，只需写一次。如果是直连机制，你需要在 JS(Web)、TypeScript(Obsidian)、Kotlin(Android)、Python(微信机器人) 中重复写 4 遍“同步 S3 文件和重建本地 SQLite”的复杂逻辑。
  - **保持“核心发动机 (Dataforge)”运转**：你之前的设计中含有翻译、食谱计算、向量增强、文本 OCR 等功能，这意味着你其实需要一个持续运转的“数据锻造炉”。
  - **避免同步冲突**：当 Obsidian 插件和 Android App 同时修改并上传同一个笔记到 S3 时，后端能够提供版本控制、锁 或者 合并策略，S3 直连往往最后导致文件覆盖丢失。

### 结论
基于 Exmemo 以大模型处理为核心基调的多前端生态，**仍然推荐采用“模式 B（统一服务器作为管理中心Hub）”进行交互。**
但是可以在具体的数据传输上做优化：例如针对大体积的文件（录音、图片），可以由后端下发基于 S3 的**预签名 URL (Presigned URL)**，使得前端能够直连 S3 下载大载荷文件，从而减轻 Server 压力。

## 多端生态下的整体系统架构探讨

在 Exmemo 这个包含多前端（Web、Android App、Obsidian 插件、微信机器人）和重度 AI 数据处理（Dataforge、语音识别、大模型翻译）的系统中，我们对“前端是否应该直连底层存储”以及“后端如何组织”进行了深入总结和探讨。

### 1. 坚持“后端 Hub（集中式 Server）”的必然性
相比于让各端 App 绕过服务端直接操作 S3/MinIO 和数据库，保留统筹一切的 ExmemoServer（后端 Hub）具有如下三个不可替代的优势：
- **避免多端平台上的“无意义重复造轮子”**：如果在客户端直连 S3，就需要在每个前端平台语言（Kotlin, TypeScript, Python, JS/Vue）中重复编写一遍繁重且复杂的“S3 文件版本双向同步、本地 SQLite 缓存建立和多维检索”逻辑。所有平台作为胖客户端也会极难维护。
- **维护 AI 数据锻造流水线 (Dataforge)**：S3 对象存储是单纯的数据舱，不具备计算能力。在 Exmemo 业务中，当新文本或录音落盘后，需要被触发自动录音转录(ASR)、大模型分析提炼(LLM)、向量化嵌入(Embeddings)等重度操作。必须依赖后端提供这台长驻运行的“发动机引擎”。
- **规避分布式的并发同步灾难**：多设备环境极易引发同一个笔记被不同前端同时修改的情境（如手机断网操作后，又在 PC 修改了同一个文件），无干预直连极大增加文件无声覆盖的风险。所有写入通过 Server 处理即可做版本锁或者冲突调和。

### 2. 高效读写最佳实践：“控制流走 Server，大载荷走直传”
虽然以 Server 统摄，但不代表所有的 I/O 开销都要经过它。现代架构采用如下优化：
- **轻量文本走 API 网关**：列表数据和纯文本从后端查询出结果后包装为轻量的 JSON，由 Server 下发前端。
- **大体积媒体走预签名直连 (Presigned URL)**：上传或下载长录音、大图片时，前端向 ExmemoServer 发送请求获得一个临时授权、单次有效的预签名链接。之后前端跳过 Server 把几十 MB 的数据直接点对点丢进 MinIO，保护了业务服务器的带宽，同时确保身份未被越权。

### 3. 后端服务模块的解耦与切割理念
确认了后端的中心地位，并不意味着把原本旧 exmemo 冗杂的业务合并写到一个巨大的“屎山代码 (Monolith)”中。面对未来复杂的功能叠加，我们规划将后端拆分为边界清晰的模块甚至独立服务：
- **API Gateway 层**：接管鉴权鉴权、多端协议适配，以及为 Obsidian、App 提供统一出口。
- **Data Core 层**：维护双轨模式的核心读写功能以及数据库索引维护。
- **Dataforge Worker/AI 计算层**：像流水线一样抽离出来，专门负责高耗时、异步的计算引擎，例如监听新接收的数据，放入队列并静默调用语音大模型或进行向量库转换。各模块松耦合，代码不再互相缠绕。
