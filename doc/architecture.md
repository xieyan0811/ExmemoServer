# 全局架构概要设计文档 (Exmemo 知识管理系统)

## 0. 设计初衷与场景引入 (核心前提)
**核心定调**：本项目始终是一个**知识管理工具（PKM）**。所有的功能演进都必须紧紧围绕知识的输入、整理、存储和输出展开，绝不能跑偏为纯聊天应用或远程电脑管控工具。

目前的构想源自以下几个核心真实场景，本架构旨在将它们无缝串联：

*   **电脑端场景（知识中枢）**：日常笔记主要围绕 Obsidian 进行。需要有一个强大的后端，不仅能实现多端坚如磐石的同步，并具备对笔记进行统一管理、统计以及辅助思考的能力。
*   **手机端场景一（深度阅读）**：也就是之前的 ExmemoReader，需要它作为一个强大的阅读工具，能把文字转成语音朗读（TTS），同时做批注笔记。这些阅读产生的笔记最终必须能统一管理，并汇入我的知识主库中（具体与电脑端笔记的结构挂靠关系待后续进一步设计）。
*   **手机端场景二（快速捕捉）**：也就是之前的 ExRecord，作为手机上的随身记事本，主要依靠强大的语音交互进行快速记录灵感、待办。内容可以随时追加、查询和修改，同样必须汇入主知识库（对接关系同上待设计）。

基于上述场景，系统后端的结构被明确划分为“绝对核心”与“衍生插件”两层：

**核心基座（笔记相关的存储与 I/O 是不可动摇的独立核心）：**
1.  **笔记底座模块**：负责笔记的存储、增删改查（CRUD）、高级搜索、Obsidian 双向同步机制等。
2.  **ASR 引擎模块**：负责将各端（特别是手机场景二）传入的语音精准转为文字，作为信息输入源。
3.  **LLM 引擎模块**：负责对录入的大量碎片化信息/笔记进行理解、自动打标签、归纳总结和建立连接，承担“思考”属性。

**衍生与外围模块（坚决以插件/独立App应用方式存在，不干扰核心）：**
如过往开发的：读论文辅助、体重测试/饮食管理、翻译、网页抓取处理、日常闲聊交互，以及某些特定维度的统计功能。它们都是建立在核心基座旁边的外挂，按需调用即可。

## 1. 产品愿景
打造一个**全能、智能、无界**的个人知识管理（PKM）工具与智能助手。
所有的知识、信息（无论是录音、阅读的原文/高亮、还是手写输入的笔记）都可以无缝汇入其中。系统能在后台自动帮你整理、统计，并在不同的知识点之间产生连接。
最终形态不仅限于传统的带界面的 App，而是可以完全基于语音交互（TTS、ASR、LLM），作为一个无界面（UI-less）或极简界面的助手出现，通过自然语言对话就能完成知识的检索、记录和整理。

## 2. 整体架构思路线：一后端、多前端
打破现有的数据孤岛，采用“**一个统一强大的后端，多个轻量/模块化的前端**”的架构体系。

*   **统一后端 (ExmemoServer + exmemo Django)**：系统大脑与数据中心。
*   **模块化多前端 (ExRecord, ExmemoReader, 笔记端等)**：数据源入口与交互端。

---

## 3. 统一后端层设计 (ExmemoServer 核心)
**战略决策：** 将未来的核心业务逻辑与数据存储重心逐步转移至 **ExmemoServer (基于 FastAPI 重构)**，全面平替原有的 `exmemo` (Django) 业务侧重。

### 3.1 核心原则与技术选型
*   **弃用 Django，拥抱 FastAPI**：服务端框架改用更轻量、对 AI 异步调用更友好的 FastAPI，以提升性能和开发体验。
*   **后端换型，数据不动**：这是一个非常关键的决定——底层 PostgreSQL 数据库及表结构（尤其是老 Django `app_dataforge` 里的 `store_entry` 等核心表）保持**完全不变**。
*   **零数据迁移**：通过 SQLAlchemy 定义新模型直接映射、挂载现有的数据库/数据表。避免在大面积框架重构的同时进行高风险的数据清洗与迁移。平滑过渡期由于存在只有个人使用的强前提，可以容忍暂时的并发兼容。

### 3.2 核心职责
*   **统一数据池 (Single Source of Truth)**：管理所有的笔记、录音转换文本、阅读记录（epub/txt/md）、高亮批注。
*   **统一 API 网关**：为所有的前端应用（浏览器、App、微信等）提供标准化、鉴权的 RESTful 或 GraphQL API。
*   **AI 大脑模块**：
    *   **ASR (语音识别)**：处理 `ExRecord` 等端传来的语音数据，转为文字。
    *   **TTS (文本转语音)**：为后续的无界面助手和阅读器提供语音播报能力。
    *   **LLM (大语言模型)**：对碎片化信息进行自动总结、标签提取、知识关联（比如自动将录音笔记与Obsidian笔记关联）。
*   **第三方同步 (Obsidian 同步引擎)**：维持与本地 Obsidian Vault 的双向同步，确保本地端与云端的统一。

## 4. 模块化多前端设计
设计要保持“不重”、模块化，做到每次开发或使用只关注小块功能，但后台数据是一体的。

### 4.1 核心模块功能
*   **录音与捕捉模块 (源自 ExRecord)**：专注快速录音、碎片化信息的快速输入。
*   **沉浸阅读模块 (源自 ExmemoReader)**：负责解析 epub, txt, markdown 等文档，支持 TTS 朗读与高亮批注。
*   **知识管理/笔记模块 (源自 exmemo web/app)**：常规的笔记浏览、编辑、整理及知识图谱查看。

### 4.2 前端演进形态（“离线优先”多端共存）
*   **坚持多端并发 (Multi-App)**：考虑到各场景功能差异巨大，不强行将它们糅合成一个笨重的超级 App。允许各个工具更专注自身特定的使用场景。
*   **离线优先 (Offline-First)**：这是极其重要的设计准则！现有的 ExRecord 和 exmemoreader 必须具备极其完整的“脱机工作”能力。即便没有后端存在，也能依靠内置的 **Room 数据库**（已在各自项目里建好）完成最基础的录音和阅读活动。
*   **同步增强 (Sync Enhancement)**：云端不再是必须，而是“增强”。网络良好时，利用后台进程（如 WorkManager）以低感知度将本地产生的高亮、录制文本汇总推送到统一的后端数据库里参与统一的分析。
*   **远期规划 (主笔记中枢 App)**：后续计划建立一个新的核心“知识汇总” Android App/终端。它专门负责浏览、组织由其他外围端（阅读端、录音端）以及电脑端Obsidian汇集过来的纯结构化笔记，甚至直接把阅读器界面的 AAR 模块复用进来。

## 5. 未来演进路线 (Roadmap)
1.  **阶段一：统一后端与数据融合 (当前)**
    *   在 `ExmemoServer` 中融合现有的 AI 能力与 Django 的数据库结构。
    *   制定统一的 API 规范，确保笔记、录音、阅读高亮的数据能在同一个数据库（或知识图谱）中共存而不冲突。
2.  **阶段二：前端服务化改造**
    *   将 `ExRecord` 和 `ExmemoReader` 等移动端应用改造为瘦客户端，数据直接推拉统一后端。
3.  **阶段三：知识连接与 LLM 深度介入**
    *   后端引入 LLM 定期/实时分析用户录入的数据，自动生成内容摘要、建立知识关联（Tagging & Linking）。
4.  **阶段四：语音助手与无界面化**
    *   在 App 端或智能手表/耳机端推出纯语音交互入口，通过说话记录灵感，通过对话获取知识库的回答。
